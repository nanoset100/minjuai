# 민주AI 온톨로지 시스템 - 쉬운 설명서

**작성일**: 2026-03-31
**목적**: 우리가 만든 것이 뭔지, 왜 대단한지를 누구나 이해할 수 있도록 기록

---

## 1. 온톨로지가 뭐야?

**한 줄 요약**: AI가 사용하는 "정책 사전"이다.

우리가 국어사전을 보고 단어의 뜻을 이해하듯이, AI도 "정책 사전"이 필요하다.
이 사전에는 대한민국 시민들이 관심 있는 258개의 정책 이슈가 정리되어 있다.

예를 들면:
- "저출산으로 인한 인력 부족" (복지)
- "전세사기 및 깡통전세 피해" (경제)
- "학교 폭력 대책" (사회)
- "미세먼지 대응" (환경)

이 258개 이슈는 7개 카테고리로 나뉜다:

| 카테고리 | 이슈 수 | 예시 |
|---------|--------|------|
| 사회 | 48개 | 학교 신설, 장애인 권리 |
| 환경 | 43개 | 농약 폐기물, 기후변화 |
| 생활 | 41개 | 소비자 보호, 교통안전 |
| 경제 | 34개 | 소상공인, 플랫폼 독점 |
| 정치 | 33개 | 외교, 선거제도 |
| 복지 | 30개 | 저출산, 고령화 |
| 미래 | 29개 | AI 로봇, 디지털 전환 |

---

## 2. 왜 만들었어?

**문제**: 시민이 "우리 동네 전세사기가 심해요"라고 제보하면, 이걸 누가 분류하지?
사람이 하면 느리고, 주관적이고, 24시간 할 수 없다.

**해결**: AI가 자동으로 분류한다!

시민이 제보를 올리면, AI가 258개 이슈와 비교해서
"이 제보는 '전세사기 및 깡통전세 피해' 이슈와 36% 유사합니다"
라고 자동으로 연결해준다.

이것이 바로 **"AI 온톨로지 매칭"**이다.

---

## 3. 어떻게 동작해? (전체 흐름)

```
시민이 앱에서 제보 작성
    "우리 동네 전세사기가 심해요"
         |
         v
    [제보 등록 버튼 클릭]
         |
         v
    서버(Railway)에 제보 저장
         |
         v
    AI가 자동으로 258개 이슈와 비교
    (이것을 "임베딩 매칭"이라고 함)
         |
         v
    "전세사기 및 깡통전세 피해" - 유사도 36%
         |
         v
    앱에 결과 표시 (BottomSheet)
         |
         v
    시민이 "맞아요" 또는 "아니에요" 투표
         |
         v
    투표 결과가 서버에 저장됨
    (AI가 맞았는지 틀렸는지 기록)
```

## 4. "임베딩"이 뭔데?

이것이 가장 신기한 부분이다.

**비유**: 도서관에서 책을 찾을 때

- **키워드 검색** = 제목에 "전세"가 들어간 책만 찾는 것
- **임베딩 검색** = 내용이 비슷한 책을 찾는 것 (제목에 "전세"가 없어도!)

AI는 모든 문장을 1,536개의 숫자로 변환한다.
이 숫자들이 문장의 "의미"를 나타낸다.

예를 들어:
- "전세사기가 심해요" → [0.12, -0.45, 0.78, ... 1,536개 숫자]
- "전세사기 및 깡통전세 피해" → [0.11, -0.43, 0.80, ... 1,536개 숫자]

두 숫자 배열이 비슷하면 → "의미가 비슷하다!" → 매칭!

이 기술을 **코사인 유사도(cosine similarity)**라고 한다.
우리 시스템에서는 35% 이상이면 "관련 있음"으로 표시한다.

## 5. 우리가 만든 것 (기술 구조)

### 5-1. 백엔드 (서버, Railway에서 돌아감)

| API | 하는 일 |
|-----|---------|
| `/api/ontology/map` | 258개 이슈 전체 목록 불러오기 |
| `/api/ontology/search?q=검색어` | 이슈 검색 |
| `/api/ontology/stats` | 통계 (총 제보 수, 매칭률 등) |
| `/api/ontology/node/{id}/reports` | 특정 이슈에 연결된 제보들 |
| `/api/districts/report/{id}/ontology` | 특정 제보의 AI 분석 결과 |
| `/api/districts/report/{id}/verify-match` | 시민 검증 투표 |

### 5-2. 데이터베이스 (Supabase)

| 테이블 | 역할 |
|--------|------|
| `ontology_nodes` | 258개 정책 이슈 저장 |
| `ontology_edges` | 이슈 간의 연결 관계 (29개) |
| `district_reports` | 시민 제보 저장 |
| `report_node_links` | 제보 ↔ 이슈 매칭 결과 + 시민 투표 |

### 5-3. Flutter 앱 (시민이 사용하는 화면)

| 화면 | 하는 일 |
|------|---------|
| 홈 대시보드 | AI 이슈 분석 현황 (총 제보, 매칭률, 평균 정확도) |
| 정책 지식맵 | 258개 이슈를 카테고리별로 탐색, 검색, 상세 보기 |
| 시민 제보 | 제보 작성 → AI 분석 → 결과 확인 → 맞아요/아니에요 투표 |

## 6. 시민 검증이 왜 중요해?

AI는 완벽하지 않다. 현재 평균 정확도는 37%다.
(100번 매칭하면 37번 정도 맞는다는 뜻)

그래서 시민이 직접 확인한다:
- "맞아요" → AI가 잘했다 (upvote +1)
- "아니에요" → AI가 틀렸다 (downvote +1)

### 솔직한 현재 상태

**지금은 투표 숫자만 DB에 쌓이고 있다. AI가 자동으로 학습하지는 않는다.**

```
시민이 "맞아요" 클릭
    ↓
DB에 숫자만 +1 됨 (verify_upvotes)
    ↓
끝. AI는 아직 이 정보를 사용하지 않음.
```

이것은 **"강화학습"이 아니다.** 정확히 말하면:

| 용어 | 뜻 | 우리 상태 |
|------|-----|----------|
| **강화학습 (RL)** | AI가 보상/벌로 행동을 스스로 개선 | 아직 아님 |
| **RLHF** | 사람 피드백으로 AI 모델을 재학습 | 아직 아님 (OpenAI 모델은 우리가 못 바꿈) |
| **데이터 수집** | 나중에 개선할 재료를 모으는 것 | **지금 이 단계** |

### 그런데 왜 이 데이터를 모아?

데이터가 충분히 쌓이면 (100건+), 코드 몇 줄로 아래 일을 할 수 있다:

1. **downvote 많은 매칭 자동 숨기기** — 가장 쉽고 즉시 효과 있음
2. **임계값 자동 조정** — upvote 비율 기반으로 0.35를 더 정확한 값으로 변경
3. **잘못된 패턴 분석** — "어떤 종류의 제보에서 AI가 자주 틀리나?" 파악

정리하면: **지금은 "AI가 똑똑해지는 재료"를 모으는 단계**이다.
재료가 쌓인 후에 활용 코드를 추가하면 그때부터 진짜 개선이 시작된다.

## 7. 개발 과정에서 배운 것

### 7-1. await showDialog 패턴 (가장 큰 버그)

제보 후 AI 결과를 보여주는 화면이 며칠간 안 나왔다.
원인: Flutter에서 다이얼로그를 닫고 다음 화면을 보여주는 순서가 잘못됐다.

```
[잘못된 방법]
다이얼로그 닫기 → 바로 다음 화면 열기 (동시에 실행되어 충돌!)

[올바른 방법]
다이얼로그 닫기를 "기다린 후" → 다음 화면 열기 (순서대로 실행)
```

Flutter의 `await showDialog` 패턴으로 해결.
**교훈**: 비동기(async) 프로그래밍에서는 "기다리기(await)"가 핵심이다.

### 7-2. data 타입 문제 (카테고리별 회색 카드)

258개 이슈 중 30개(원본)는 데이터가 "객체"로 저장되어 있었고,
228개(나중에 추가)는 데이터가 "문자열"로 저장되어 있었다.

```
원본 30개: data = {severity: "high"}         ← 객체 (접근 가능)
추가 228개: data = '{"source": "시민단체"}'  ← 문자열 (접근하면 에러!)
```

객체인지 문자열인지 먼저 확인하는 방어 코드로 해결.
**교훈**: 외부 데이터는 항상 타입을 확인해야 한다.

### 7-3. 임베딩 제외 (1.4MB → 366KB)

원래 API가 258개 이슈 + 임베딩(1,536개 숫자 x 258개)을 모두 보냈다.
앱에서는 임베딩이 필요 없으므로 제외 → 데이터 크기 74% 감소!
**교훈**: 필요한 데이터만 주고받아야 앱이 빨라진다.

## 8. 3개 AI 크로스 리뷰

이 시스템을 만들기 전에, 3개의 다른 AI에게 설계를 검토받았다:

| AI | 점수 | 가장 유용한 피드백 |
|----|------|------------------|
| **Grok** (xAI) | 8.5/10 | BottomSheet UX 구조 제안 |
| **Claude Sonnet** (Anthropic) | 9/10 | 유사도 0.35 임계값 발견 (핵심!) |
| **Gemini** (Google) | 7.5/10 | 텍스트 오버플로우 방지 |

Claude Sonnet이 발견한 "0.35 임계값"이 가장 중요했다.
우리 데이터의 유사도가 0.32~0.40 범위인데,
기본값(0.5)을 사용했으면 아무 결과도 안 나올 뻔했다!

## 10. 앞으로 가능한 것들

이 시스템의 진짜 가치는 **데이터가 쌓일수록** 나타난다.
단, 데이터만 쌓인다고 자동으로 좋아지는 건 아니다. **활용 코드를 추가해야** 한다.

### 데이터만 쌓이면 할 수 있는 것 (코드 추가 필요)

1. **시민 제보가 100건, 1000건** 쌓이면 → 어떤 이슈가 가장 뜨거운지 통계 분석
2. **검증 투표가 쌓이면** → downvote 많은 매칭 숨기기, 임계값 조정 (현재 고정 0.35)
3. **지역별 데이터가 쌓이면** → "서울 강남구는 교육, 부산 해운대는 환경" 같은 지역별 이슈 지도

### 별도 개발이 필요한 것 (큰 작업)

4. **AI 모델 자체 개선** → Fine-tuning 또는 다른 임베딩 모델 적용 (정확도 37% → 70%+ 목표)
5. **실시간 트렌드** → "이번 달 급상승 이슈: 전세사기" 같은 시계열 분석
6. **국회의원별 연결** → "이 의원의 지역구 핫이슈" → 의정활동 평가에 활용

**핵심**: 지금 만든 것은 "기반 인프라"다.
시민의 투표 데이터는 모이고 있고, 이것을 활용하는 코드를 추가하면
단계적으로 시스템이 개선된다. 자동이 아니라, **의도적으로 개선하는 것**이다.
